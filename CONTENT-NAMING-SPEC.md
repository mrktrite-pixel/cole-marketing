# COLE Content Naming Specification

**Created:** May 4 2026 (Block 2 Task 2.0a)
**Status:** Locked. All COLE content assets — generated by any platform station — follow this spec.

---

## 1. Why this spec exists

When the bees produce videos, carousels, images, or any other content asset, the file needs to land somewhere structured. Without a naming spec:

- Files land in random storage paths (waste, hard to find)
- No version tracking when Scientist Bee proposes V2s (collisions)
- Cleaners (Madame's K15, K17) can't query what to delete
- Cross-site replication breaks (taxchecknow vs viabilityindex collisions)
- Audit trail lost (which bee made this? when? from what research?)

This spec gives every asset a globally unique, human-readable, queryable identifier from the moment it's created.

---

## 2. The canonical format

```
[site]-[product_short]-[platform_short]-[YYYYMMDD]-[seq][ver]
```

**Examples:**

```
taxchecknow-au19-tt-20260511-001a
taxchecknow-au01-ig-20260512-002b
viabilityindex-fr03-yt-20260601-001a
cryptochecknow-cr12-x-20260615-003a
```

**Field-by-field:**

| Field | Value | Constraint |
|---|---|---|
| `site` | The COLE site identifier | lowercase, kebab if needed; e.g. `taxchecknow`, `viabilityindex` |
| `product_short` | Short product key | lowercase, e.g. `au19` (from `au-19-frcgw-clearance-certificate`) |
| `platform_short` | 2-4 char platform code | see Section 3 |
| `YYYYMMDD` | Generation date in WST | ISO date, no separators; e.g. `20260511` for May 11 2026 |
| `seq` | 3-digit sequence number for that day | `001`, `002`, `003`. Resets per (site, product, platform, day). |
| `ver` | Single lowercase letter | `a` = V1, `b` = V2 (Scientist Bee redo), `c` = V3, etc. |

**Total length:** 30-40 chars depending on site and product names.

## 3. Platform short codes

Two-to-four character codes used in asset names and storage paths:

| Platform | Short code |
|---|---|
| TikTok | `tt` |
| Instagram | `ig` |
| YouTube Shorts | `yts` |
| YouTube Long-form | `ytl` |
| X / Twitter | `x` |
| LinkedIn | `li` |
| Threads | `th` |
| Reddit | `rd` |
| Pinterest | `pn` |
| Facebook | `fb` |
| Bluesky | `bs` |

For the same platform with different content types, prefix with the type:
- `ig-reel` (Instagram Reel)
- `ig-carousel` (Instagram carousel)
- `ig-story` (Instagram story)

When ambiguous, full short code goes in. When platform implies content type (TikTok = always video), simple code is fine.

---

## 4. Sequence numbers

`seq` resets every day per `(site, product, platform)` triple.

**Example:** TikTok produces 3 videos for AU-19 on May 11 2026:
```
taxchecknow-au19-tt-20260511-001a
taxchecknow-au19-tt-20260511-002a
taxchecknow-au19-tt-20260511-003a
```

Next day, sequence restarts:
```
taxchecknow-au19-tt-20260512-001a
```

This means seq is small (rarely > 005 in practice) and predictable.

**Why per-day reset:** combined with the date, every asset is unique within day. No need for per-product running counter that grows unbounded.

---

## 5. Version letters

`ver` tracks revisions of the same asset. The version letter increments only when Scientist Bee (or operator) proposes a redo of the SAME content.

**Example:** TikTok video about AU-19 generated May 11, performed poorly (low completion rate). Scientist Bee proposes V2 with hook variant:
```
taxchecknow-au19-tt-20260511-001a   ← original (V1)
taxchecknow-au19-tt-20260511-001b   ← Scientist V2, same date as parent
```

V2 keeps the same date (parent's generation date) so you can see the lineage at a glance.

If Scientist proposes V3 too:
```
taxchecknow-au19-tt-20260511-001a
taxchecknow-au19-tt-20260511-001b
taxchecknow-au19-tt-20260511-001c
```

Version letters are local to the (site, product, platform, date, seq) tuple.

**Tracking lineage:** `content_assets.parent_asset_id` points to the previous version's UUID. The naming convention is for human/file-level identification; database parent_asset_id is for queries.

---

## 6. Storage path layout

Every generated file lives at a predictable path:

```
storage/[site]/[product_short]/[platform_short]/[YYYYMM]/[asset_name].[ext]
```

**Examples:**
```
storage/taxchecknow/au19/tt/202605/taxchecknow-au19-tt-20260511-001a.mp4
storage/taxchecknow/au19/tt/202605/taxchecknow-au19-tt-20260511-001b.mp4
storage/taxchecknow/au19/tt/202605/taxchecknow-au19-tt-20260513-001a.mp4
storage/taxchecknow/au01/ig/202605/taxchecknow-au01-ig-20260512-001a.mp4
storage/viabilityindex/fr03/ytl/202606/viabilityindex-fr03-ytl-20260601-001a.mp4
```

**Why this structure:**

- **Site at root:** scales to 100+ sites without conflict
- **Product next:** all assets for one product co-located (easy audit)
- **Platform next:** per-platform sweep operations are simple
- **YYYYMM bucket:** monthly cycle is the cleaner's natural sweep boundary
- **Asset name as filename:** filename alone identifies the asset uniquely
- **Extension last:** `.mp4`, `.png`, `.pdf`, `.txt` etc. per asset type

**Critical for Madame K15:** Sweep "everything in `storage/*/*/*/202604/`" deletes all April 2026 content older than current month. Single glob, no metadata join needed.

## 7. The content_assets table (specced in Task 2.0b)

Every generated asset gets a row. Schema:

```sql
CREATE TABLE content_assets (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  asset_name TEXT NOT NULL UNIQUE,    -- 'taxchecknow-au19-tt-20260511-001a'
  
  site TEXT NOT NULL,
  product_key TEXT NOT NULL,          -- full key, e.g. 'au-19-frcgw-clearance-certificate'
  product_short TEXT NOT NULL,        -- e.g. 'au19'
  platform TEXT NOT NULL,             -- e.g. 'tiktok'
  platform_short TEXT NOT NULL,       -- e.g. 'tt'
  asset_type TEXT NOT NULL,           -- 'video' / 'carousel' / 'image' / 'thread'
  
  generation_date DATE NOT NULL,      -- 2026-05-11 (WST)
  sequence_number INT NOT NULL,       -- 1, 2, 3
  version_letter TEXT NOT NULL,       -- 'a', 'b', 'c'
  parent_asset_id UUID REFERENCES content_assets(id),  -- V2 points to V1's row
  
  storage_path TEXT NOT NULL,         -- 'storage/taxchecknow/au19/tt/202605/...'
  storage_url TEXT,                   -- public URL or signed URL
  bytes BIGINT,                       -- size for K15 cost-tracking
  
  -- Provenance (which bees produced this)
  research_job_id UUID,               -- N1 output that informed this
  strategy_job_id UUID,               -- N2 output that picked the hook
  adapter_job_id UUID,                -- N3 output that wrote the script
  manager_check_id UUID,              -- N4 quality gate result
  publisher_job_id UUID,              -- N5 result (or NULL if manual)
  
  -- Lifecycle status
  status TEXT NOT NULL CHECK (status IN (
    'draft',           -- N3 wrote it, N4 hasn't checked yet
    'approved',        -- N4 passed, awaiting publish
    'rejected',        -- N4 failed, needs redo
    'manual_pending',  -- approved, waiting on operator manual upload (warm-up phase)
    'published',       -- live on platform
    'superseded',      -- newer version exists, archive candidate
    'archived'         -- moved to cold storage, K15 cleanup eligible
  )),
  
  published_at TIMESTAMPTZ,
  archive_eligible_at TIMESTAMPTZ,    -- when K15 can sweep this
  
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  
  UNIQUE (site, product_short, platform_short, generation_date, sequence_number, version_letter)
);

CREATE INDEX idx_content_assets_site_status ON content_assets(site, status);
CREATE INDEX idx_content_assets_archive_eligible ON content_assets(archive_eligible_at) 
  WHERE status IN ('superseded', 'archived');
CREATE INDEX idx_content_assets_platform_date ON content_assets(platform, generation_date);
```

The unique constraint on (site, product_short, platform_short, generation_date, sequence_number, version_letter) enforces the naming spec at the database level.

## 8. Asset status lifecycle

```
                                  ┌──────────────┐
                                  │   rejected   │ ← N4 fails
                                  └──────────────┘
                                         ↑
[N3 generates] → draft ──N4 checks──→ approved ──N5 publishes──→ published
                                         ↓                            ↓
                                   manual_pending                superseded
                                   (warm-up only)                (V2 ships)
                                         ↓                            ↓
                                  [operator uploads]              archived
                                   → published                   (after grace period)
                                                                      ↓
                                                              [K15 deletes file]
```

**Transitions:**

| From | To | Trigger |
|---|---|---|
| (none) | `draft` | N3 generates content |
| `draft` | `approved` | N4 quality check passes |
| `draft` | `rejected` | N4 quality check fails |
| `approved` | `manual_pending` | N5 in warm-up mode (manual_handoff) |
| `approved` | `published` | N5 in auto mode posts to API successfully |
| `manual_pending` | `published` | Operator marks "uploaded" in Soverella |
| `published` | `superseded` | New version (V2) ships and replaces this in feed |
| `superseded` | `archived` | After grace period (default 7 days) |
| `archived` | (deleted) | K15 sweeper removes file + row |

K15 only sweeps rows where `status IN ('archived')` AND `archive_eligible_at < NOW()`.

## 9. The lib/content-naming.ts helper (specced in Task 2.0c)

Single TypeScript module with these exports:

```typescript
// lib/content-naming.ts (Block 2 Task 2.0c)

export interface AssetIdentity {
  site: string;
  productShort: string;
  platformShort: string;
  generationDate: Date;     // WST, normalized to date
  sequenceNumber: number;
  versionLetter: string;
}

/**
 * Generate the next available asset name for a (site, product, platform, date) tuple.
 * Queries content_assets to find the next sequence number.
 */
export async function generateAssetName(
  site: string,
  productKey: string,
  platform: string,
  date?: Date  // defaults to today WST
): Promise<{ assetName: string; identity: AssetIdentity }>;

/**
 * Parse an asset name back into its components.
 * Throws if the name doesn't match the canonical format.
 */
export function parseAssetName(assetName: string): AssetIdentity;

/**
 * Build the full storage path for an asset.
 */
export function buildStoragePath(
  identity: AssetIdentity,
  extension: string  // 'mp4', 'png', etc.
): string;

/**
 * For a given parent asset, generate the next version letter.
 * Used by Scientist Bee when proposing V2/V3.
 */
export async function nextVersionLetter(
  parentAssetId: string
): Promise<string>;

/**
 * Convert full product key to short form.
 * 'au-19-frcgw-clearance-certificate' → 'au19'
 */
export function productKeyToShort(productKey: string): string;

/**
 * Convert platform name to short code.
 * 'tiktok' → 'tt', 'instagram' → 'ig', 'youtube-long' → 'ytl'
 */
export function platformToShort(platform: string): string;
```

Used by every adapter bee. Single source of truth. No string-template duplication across bees.

## 10. Monthly cycle for Madame's cleaners

Madame's K15 (Storage Sweeper) runs Sunday 4am AWST. Logic:

```sql
-- K15 query: find archive-eligible assets
SELECT id, asset_name, storage_path
FROM content_assets
WHERE status IN ('archived')
  AND archive_eligible_at < NOW()
LIMIT 100;

-- For each, delete file at storage_path, then DELETE row
```

Monthly cycle works because the storage path includes `YYYYMM`:

- Today is May 4 2026 (current month: 202605)
- Last month boundary: April 2026 (202604)
- Two months ago: March 2026 (202603)

K15 default policy: assets in `published` status remain forever (they're live, deleting breaks the platform link). Only `archived` rows get swept.

But the YYYYMM bucket lets cleanups operate per-month: "find all archived rows from `storage/*/*/*/202604/`" is one path-level glob.

## 11. Integration pattern for new platform stations

When a new platform station builds (M-station for Instagram, L-station for YouTube, etc.):

1. Add platform short code to `platformToShort()` in `lib/content-naming.ts`
2. M3 / L3 / K3 / Q3 adapter bee imports `generateAssetName()` 
3. Adapter writes file to `buildStoragePath()` location
4. Adapter inserts content_assets row with status='draft'
5. Manager bee (M4 / L4 / etc.) updates row to 'approved' or 'rejected'
6. Publisher bee (M5 / L5 / etc.) updates row to 'manual_pending' or 'published' based on warm-up state

No naming logic in the bee itself — all delegated to the helper. The bee only fills in platform-specific fields.

## 12. RLS (Row-Level Security)

Per Locked Rule #5 (multi-site data isolation):

```sql
-- Every query must filter by site
ALTER TABLE content_assets ENABLE ROW LEVEL SECURITY;

CREATE POLICY content_assets_per_site ON content_assets
  FOR ALL
  TO authenticated
  USING (site = current_setting('app.current_site', true))
  WITH CHECK (site = current_setting('app.current_site', true));
```

The application sets `app.current_site` per request. Bees running for taxchecknow only see taxchecknow rows. Cross-site contamination impossible at DB level.

## 13. What this spec does NOT cover

- Asset binary content itself (videos, images) — that's Supabase Storage / S3
- Per-platform metadata (TikTok privacy fields, YouTube title/description) — those live in `content_jobs.output_data.[platform]_published_meta` per Chat A Rule 9
- Performance metrics (views, completion, conversions) — that's `content_performance` table
- Scheduling — that's `campaign_calendar`

Content_assets is the **lifecycle registry** of generated assets. It's the bridge between content_jobs (what got planned) and content_performance (what got measured).

## 16. Examples for clarity

### TikTok video pipeline (full example)

May 11 2026 WST, Gary Mitchell character producing TikTok video for AU-19:

1. **N1 Research** writes `content_jobs.output_data.tt_research = {...}` (no asset yet)
2. **N2 Strategy** picks hook, writes `tt_strategy = {...}` (no asset yet)
3. **N3 Adapter** generates script, calls Blotato to render MP4. **Creates content_assets row:**
   - `asset_name = 'taxchecknow-au19-tt-20260511-001a'`
   - `site = 'taxchecknow'`
   - `product_key = 'au-19-frcgw-clearance-certificate'`
   - `product_short = 'au19'`
   - `platform = 'tiktok'`
   - `platform_short = 'tt'`
   - `asset_type = 'video'`
   - `generation_date = '2026-05-11'`
   - `sequence_number = 1`
   - `version_letter = 'a'`
   - `storage_path = 'storage/taxchecknow/au19/tt/202605/taxchecknow-au19-tt-20260511-001a.mp4'`
   - `status = 'draft'`
4. **N4 Manager** runs 9-point checklist. Passes. Updates `status = 'approved'`.
5. **N5 Publisher** reads warm-up guard. Today is May 11. TikTok warm-up complete May 31. So mode is `manual_handoff`. Updates `status = 'manual_pending'`. Notifies operator via Soverella.
6. **Operator** opens Soverella's "Awaiting Manual Upload" tab, sees the row, downloads the MP4, manually uploads to TikTok app, taps "Mark as posted" in Soverella. UPDATE sets `status = 'published'` and `published_at = NOW()`.
7. **Three weeks later (June 1+):** TikTok account is now warm. Same pipeline next time hits N5 in `auto` mode, status goes straight to `published`.

### Scientist V2 example

July 4 2026: K-station notices AU-19 TikTok video from May 11 has 12% completion rate (target: 70%). Scientist Bee proposes V2 with new hook (drop_off at 5-12s = PROMISE FAILURE per HIVE 3 doctrine).

1. Scientist writes proposal to `agent_log` for operator approval
2. Operator approves in Soverella
3. N3 reruns with new hook prompt. Calls helper:
   - `nextVersionLetter('uuid-of-001a-row')` → returns `'b'`
   - Creates new row: `taxchecknow-au19-tt-20260511-001b`
   - `parent_asset_id = <UUID of 001a>`
   - `generation_date = '2026-05-11'` (parent's date, preserves lineage)
4. N4 → N5 normal flow
5. When 001b publishes, the existing 001a row updates to `status = 'superseded'`, `archive_eligible_at = NOW() + INTERVAL '7 days'`
6. After 7 days, K15 sweeps storage file, then deletes the row
